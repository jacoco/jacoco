Contribute to JaCoCo

From the very beginning JaCoCo was managed as a free and open project. All work artifacts like code, build results, discussions on mailing lists, the bug tracker and our wiki are publicly accessible. Many great features and bug fixes have been contributed by the community.

There are many ways to contribute to the JaCoCo project:

  • Create a well-structured bug report using the provided template that also comes with a executable reproducer.
  • Provide a features request that comes with enough context information so everybody can understand the purpose and value of the feature.
  • Share insights on our mailing list, especially if you have experience on specific platforms like mobile development, application servers or fancy build systems.
  • Contribute to documentation, even if it's only about fixing typos.
  • Test new features and snapshot builds and share your discoveries.
  • Provide a idea how to fix a known bug or pending feature request.
  • Provide a fix for a known or new bug as a pull request.
  • Provide a new feature as a pull request.

The following topics provides a guide for successful contributions.

Development Setup

Our standard development setup is based on the Eclipse IDE and is described here. All contributions need to be in-line with our development conventions. Also some information about the implementation design can be helpful to understand JaCoCo's internal.

Process

All code contributions (even the contributions by the core developers) are managed by GitHub pull requests. Typically a pull request references an open issue (bug or feature request). Each pull request has to be complete in terms of features, testing and documentation before it can be merged into master. We consider the master branch as shippable at any point in time. This of course means that all builds on all supported JDKs must be green. Every PR is automatically built and the result is reported to the GitHub pull request.

Pull requests will be reviewed and discussed by the JaCoCo project team. Once we decided to adopt a contribution we will help to get the everything cleaned up before the merge. During this process more commits might be added to address issues. If you allow edits from maintainers the project team can do minor cleanups and the paperwork (e.g. change log) for external contributors directly.

Once the PR is finalized all commits are squashed by the project maintainers and added to master as a single commit.

Before you start a bigger contribution please check whether we have an open issue for this (or open a new issue first) to make sure the JaCoCo project considers this in scope and the approach fits in the overall architecture. In doubt please get in touch through our mailing list

Scope

Keep your contributions as specific as possible. One thing at a time. "I fixed this and that" style pull requests typically are very hard to review and discuss as a single controversial aspect might kill the whole pull request.

Also JaCoCo project is a code coverage library with some tool integration. We will not extend the scope to other domains (profiling, code metrics, etc.).

Technical Depth

The JaCoCo team is committed to support the project in the long run. Therefore we do not accept any increase of technical depth in our code base. Contributions with questionable design, missing tests or poor code quality need to be reworked before they can be adopted. "We fix that later" is generally no option in the JaCoCo project.

Please make sure your contribution follows the JaCoCo project's development conventions.

Test Coverage

JaCoCo is developed in a fully test-driven manner. We're nearly at 100% unit test coverage and have many automated integration tests to validate our features with every build. We expect every new feature to come with full test coverage. And of course we assume that all tests are green and stable before we review pull requests.

If you're unsure how to test certain aspects please come to our mailing list to discuss a solution.

Legal Issues

We want to keep JaCoCo consumable as free and open software. For this we need to stick with some basic rules for code contributions:

  • Contributors (and if applicable their employer) need to agree that JaCoCo and so all contributions are published under EPL 1.0.
  • Whatever is contributed has to be personal work or otherwise needs proper attribution to its original authors.
